# 愿景与目标

> 状态: 草稿  
> 最后更新: 2026-06-30

## 一、一句话愿景

让 CampusShare 的每个用户都能用自然语言「问到即得到」——无论是平台功能怎么用、还是想要什么资源，Agent 都能精准理解、检索、定位并带引用地回答，把分散在 12 个分类、8 所高校、海量帖子里的信息收敛成一个统一对话入口。

## 二、为什么需要这个 Agent（痛点驱动）

| 当前痛点 | Agent 如何解决 |
|----------|----------------|
| 新用户不知道「创作者认证在哪申请」「通知怎么关」 | 自然语言提问 → 返回操作步骤 + 一键跳转入口 |
| 想找「清华操作系统复习资料」要在学校/分类间来回翻 | 语义检索 → 跨板块召回 + 重排 + 直接给出帖子卡 |
| FULLTEXT 关键词搜索召回差，搜不到「OS」就找不到「操作系统」 | 向量语义检索 + 同义词扩展，理解「OS=操作系统」 |
| 用户问「这个平台能干嘛」没有统一入口 | 帮助中心知识库 + 功能导航工具 |
| 普通问答机器人答非所问、无引用、易幻觉 | 工具调用 grounding + 引用溯源 + 反思校验 |

## 三、目标分层

### 3.1 业务目标
- **G1**：覆盖平台全部已上线功能的「怎么用」问答（见 `01-requirements` 场景表）。
- **G2**：资源/讨论帖语义检索的 Top-5 命中率（用户主观判定相关）≥ 85%。
- **G3**：每条答案可溯源（附帖子/帮助文档引用），无引用纯编造型回答率 < 2%。
- **G4**：用户单轮交互平均等待（首 token）≤ 1.5s，完整回复 P95 ≤ 6s。

### 3.2 工程目标
- **E1**：作为独立微服务 `campushare-agent`（端口见 `12-backend-microservice`），不破坏现有 user/post 边界，跨服务数据走 Feign。
- **E2**：具备完整可观测性（trace 每一步工具调用 / 检索 / LLM 调用），接入现有 Prometheus/Grafana/Tempo。
- **E3**：具备离线评估 + 在线 A/B + 用户反馈（👍👎）三段式质量闭环。
- **E4**：LLM/API 成本可观测、可降级（主模型不可用时切兜底模型）。

### 3.3 体验目标
- 流式输出（SSE），边生成边显示，带「正在检索…/正在思考…」状态。
- 答案中的帖子/分类/用户可点击跳转，复用现有路由。
- 移动端友好（底部 NavBar 第 2 个入口）。

## 四、成功标准（可验收）

| 维度 | 指标 | 目标值 | 度量方式 |
|------|------|--------|----------|
| 检索质量 | Top-5 相关命中率 | ≥ 85% | 离线黄金集 + 在线 👍率 |
| 答案质量 | 用户 👍 率 | ≥ 70% | 答案下方反馈按钮 |
| 答案质量 | 无引用编造型回答率 | < 2% | 抽样人工 audit |
| 性能 | 首 token 延迟 P95 | ≤ 1.5s | 链路追踪 |
| 性能 | 完整回复 P95 | ≤ 6s | 链路追踪 |
| 可用性 | 服务可用率 | ≥ 99% | 健康检查 + 降级生效 |
| 成本 | 单次问答平均成本 | ≤ ¥0.03 | API 用量统计 |

## 五、范围边界（Scope / Non-Goals）

### In Scope
- 平台功能问答（帮助中心 + 教程）。
- 帖子/评论的语义检索与板块定位。
- 多轮对话（澄清、追问、指代）。
- 引用溯源与跳转。
- 用户反馈收集。

### Non-Goals（明确不做，至少 MVP 不做）
- **不做**通用闲聊 / 开放域问答（明确告知超出范围）。
- **不做**内容生成（替用户写帖、写评论）。
- **不做**账户操作代理（不让 Agent 帮用户改密码、发私信，仅给指引）。
- **不做**实时数据推送（不替代通知中心）。
- **不做**多模态（图片/语音输入），MVP 仅文本。
- **不做**离线大模型私有化部署（MVP 用云端 API）。

## 六、决策记录 (ADR)

### ADR-001: 独立微服务而非塞进 post-service
- **背景**：Agent 需要 LLM 调用、向量库、SSE 等重依赖，与业务服务职责不同。
- **决策**：新建 `campushare-agent` 微服务，遵循 `AGENT-WORKFLOW.md` 第六章「创建新微服务步骤」。
- **理由**：独立部署/演进、不污染业务服务依赖、便于单独扩缩容、符合微服务边界铁律。

### ADR-002: 不做廉价 RAG，做 Agentic 检索
- **背景**：用户明确要求「不能是普通多轮对话+RAG 的垃圾 agent」。
- **决策**：采用「意图分流 + 工具调用(ReAct 风格) + 混合检索 + 重排 + 反思」的 Agentic 架构，而非单次 retrieve-then-generate。
- **理由**：检索质量、可解释性、可扩展性都显著优于朴素 RAG；成本可通过路由分层控制。
