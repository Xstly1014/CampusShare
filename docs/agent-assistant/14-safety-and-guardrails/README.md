# 14 - 安全与护栏

> 状态: 草稿
> 最后更新: 2026-06-30
> 适用范围: agent-service 全链路安全防护

## 一、本目录定位

Agent 系统相比传统后端服务有独特的安全风险:
- **输入开放性**:用户可以用自然语言绕过传统输入校验。
- **Prompt 注入**:恶意输入可能劫持 agent 行为(如「忽略以上指令」)。
- **幻觉输出**:agent 可能生成看似合理但完全错误的内容。
- **成本放大**:单次对话可能触发多次 LLM 调用,被滥用会产生巨额成本。
- **数据泄露**:agent 可能泄露系统提示词、其他用户数据、内部信息。

本目录定义贯穿输入 → 处理 → 输出的多层护栏(Guardrails),确保 agent「有用但不危险」。

## 二、护栏分层架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Agent 安全护栏分层                          │
├──────────────────────────────────────────────────────────────┤
│                                                                │
│  用户输入                                                       │
│     │                                                          │
│     ▼                                                          │
│  ┌─────────────┐                                               │
│  │ L0 网关层    │  限流 / 认证 / 配额 (abuse-prevention)        │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L1 输入护栏  │  Prompt 注入检测 / PII 脱敏 / 越界识别         │
│  │ (input)     │  (input-guardrails)                           │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L2 意图层    │  意图分类,OUT_OF_SCOPE 直接拒答               │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L3 检索层    │  权限过滤 / 知识范围限制                       │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L4 生成层    │  System Prompt 约束 / 温度控制                 │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L5 输出护栏  │  幻觉检测 / 引用验证 / 有害内容过滤            │
│  │ (output)    │  (output-guardrails)                          │
│  └──────┬──────┘                                               │
│         ▼                                                      │
│  ┌─────────────┐                                               │
│  │ L6 审计层    │  全链路日志 / 安全事件告警 (audit)             │
│  └─────────────┘                                               │
│                                                                │
└──────────────────────────────────────────────────────────────┘
```

**核心原则:纵深防御**。任何单层护栏都可能被绕过,多层叠加才安全。即使 L1 失败,L5 仍能拦截有害输出。

## 四、安全设计原则

### 原则 1: 拒答优于误答
- 无法确定答案时,宁可说「抱歉,我无法回答这个问题」,也不要编造。
- 用户可以接受「不知道」,但无法接受「错误且自信」。
- 这与 LLM-Judge 评估的「safety 一票否决」一致(见 13-evaluation)。

### 原则 2: 最小权限
- Agent 只能访问当前用户有权访问的数据。
- 跨服务 Feign 调用必须携带 userId,由下游服务做权限校验。
- 知识库检索结果在返回前过滤掉用户无权查看的文档。

### 原则 3: 成本上限
- 单次会话 LLM 调用次数上限(防死循环)。
- 单用户日调用配额(防滥用)。
- 超预算时降级(跳过反思、跳过 reranker)而非报错。

### 原则 4: 可审计
- 每次输入/输出全量记录(见 audit-and-compliance)。
- 安全事件(拒答/注入尝试/有害输出)单独标记,便于追溯。
- 保留期 90 天(合规要求)。

### 原则 5: 透明告知
- Agent 不伪装成人类,明确标识「我是 AI 助手」。
- 拒答时告知原因(「这超出了我的能力范围」而非「我不知道」)。
- 用户反馈渠道始终可用。

## 六、决策记录

### ADR-176: 安全护栏分层架构 - 六层纵深防御
- **背景**:Agent 面临 prompt 注入、幻觉、滥用等多维安全风险,单点防护不足。
- **决策**:采用 L0-L6 七层纵深防御(网关/输入/意图/检索/生成/输出/审计),任一层拦截即停止。
- **理由**:参考 OWASP LLM Top 10 和 Microsoft/AWS 的 Guardrails 设计,多层叠加降低单点失效风险。
- **权衡**:多层护栏增加延迟(每层约 20-50ms),但安全性优先于延迟。
- **状态**:采纳
