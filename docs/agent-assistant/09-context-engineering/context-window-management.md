# 上下文窗口与 Token 预算管理

> 状态: 草稿
> 最后更新: 2026-06-30

## 一、目标

在 DeepSeek-V3 的 128K 窗口下，把单轮 input 控制在 **≤8K tokens**（典型 4-6K），兼顾成本、延迟、回答质量。任何超预算情况都必须有降级路径，而不是直接报错。

## 二、上下文分层结构

按"重要性 + 可压缩性"分 5 层，自顶向下装入：

| 层 | 内容 | 是否永驻 | 典型 token | 可压缩性 |
|----|------|---------|----------|---------|
| L0 | System Prompt（平台级 + 任务级） | ✅ 永驻 | 800-1200 | 不可压缩 |
| L1 | 用户画像摘要（长期记忆） | 按需装载 | 200-400 | 可裁剪 |
| L2 | 工具/函数定义（Function Calling schema） | ✅ 永驻 | 400-600 | 不可压缩 |
| L3 | 检索结果 + 工具调用结果 | 按需装载 | 1500-3000 | 高（按 RRF 分截断） |
| L4 | 对话历史（短期记忆窗口） | ✅ 永驻最近 N 轮 | 1000-2000 | 中（可压缩摘要） |
| L5 | 当前用户输入 + 意图分类结果 | ✅ 永驻 | 50-500 | 不可压缩 |

总和预算：**8000 tokens**。L0+L2+L5 ≈ 2000 是硬性下限，剩余 6000 在 L1/L3/L4 间动态分配。

## 三、Token 预算分配策略

### 3.1 静态分配（默认）

```
L0 System Prompt     : 1000  (固定)
L1 用户画像          :  300  (固定预留)
L2 工具定义          :  500  (固定)
L3 检索/工具结果     : 3000  (主预算池)
L4 对话历史          : 2500  (固定预留)
L5 用户输入          :  700  (按实际)
─────────────────────────────
合计                  : 8000
```

### 3.2 动态再分配

根据意图与阶段动态调整 L3/L4：

| 意图/阶段 | L3 检索 | L4 历史 | 说明 |
|----------|--------|--------|------|
| HOW_TO（知识库命中） | 4000 | 1500 | 知识片段优先，历史可压缩 |
| SEARCH（帖子检索） | 3500 | 2000 | 帖子摘要 + 历史指代消解 |
| NAVIGATE | 1000 | 3000 | 几乎不需要检索，靠历史 |
| CLARIFY（追问） | 500 | 4000 | 历史是核心 |
| 反思阶段（R1） | 4000 | 1500 | 反思需要看全部证据 |
| 多 Agent 综合 | 4000 | 1500 | 各子 Agent 输出汇总 |

### 3.3 超预算降级链

当实际 token 超过预算时，按顺序触发：

1. **L3 截断**：按 RRF 分从低到高丢弃检索块，保留 Top-K（K 自适应）。
2. **L4 压缩**：触发历史摘要（见 [context-compression.md](./context-compression.md)）。
3. **L1 裁剪**：用户画像只保留"偏好类"，丢弃"历史行为类"。
4. **L2 旁路**：若当前轮不需要工具调用，移除工具定义（节省 500 token）。
5. **硬上限兜底**：若仍超 12000，截断 L4 到最近 2 轮，并记录 `context_truncated=true` 到会话日志。

## 四、Token 计数实现

### 4.1 估算器

- 使用 `tiktoken`（Python）或 `jtokkit`（Java，`com.knuddels:jtokkit`）做本地估算，避免每次都调 API。
- DeepSeek-V3 兼容 OpenAI `cl100k_base` 编码，估算误差 <5%。
- 中文 1 字 ≈ 1.3 token，英文 1 词 ≈ 1.3 token，作为快速估算的 fallback。

### 4.2 计数埋点

每次组装完上下文，记录：

```json
{
  "session_id": "...",
  "turn_id": 7,
  "layer_tokens": {
    "L0": 1024, "L1": 280, "L2": 512,
    "L3": 2890, "L4": 2100, "L5": 86
  },
  "total_input_tokens": 6892,
  "budget": 8000,
  "truncated": false,
  "truncation_reason": null
}
```

写入 `agent_context_snapshots` 表，供后续复盘与离线分析。

## 五、上下文缓存命中

### 5.1 DeepSeek Context Cache

- DeepSeek-V3 支持 prefix cache：相同前缀的请求只计费 0.1×。
- **L0 + L2 + L1（用户画像稳定部分）** 放在最前面，作为缓存前缀。
- 同一用户同一会话的连续请求，前缀命中率目标 ≥90%。

### 5.2 前缀稳定性约束

- L0 System Prompt 在一个 prompt 版本内**字节级固定**，任何空格/标点变更都会击穿缓存。
- L1 用户画像摘要采用"固定字段 + 固定顺序"，画像更新时通过版本号控制（不是每次都改文本）。
- L4 历史按"旧→新"追加，旧部分不变。

### 5.3 前缀击穿检测

监控 `cache_hit_rate` 指标，若单会话内 <70%，触发告警并排查：
- 是否 System Prompt 被动态拼接了时间戳？
- 是否用户画像每轮都在变？
- 是否 Few-shot 示例被随机采样导致顺序变化？

## 六、与现有架构的接入

### 6.1 上下文组装器 ContextAssembler

新增 `ContextAssembler` 组件（在 agent-service 内）：

```
输入: sessionId, userQuery, intent, retrievalResults, toolResults
输出: { messages: ChatMessage[], tokenCount: int, snapshot: ContextSnapshot }
```

调用顺序：
1. 从 Redis 取会话状态 + 短期记忆窗口。
2. 从 MySQL 取用户画像摘要（带 TTL 的本地缓存）。
3. 按意图选择分配策略。
4. 按层装入并计数。
5. 超预算时按降级链压缩。
6. 输出 messages + 快照。

### 6.2 与 Prompt 模板的协作

- Prompt 模板里用 `{{USER_PROFILE}}`、`{{RETRIEVAL_CONTEXT}}`、`{{CONVERSATION_HISTORY}}` 占位符。
- ContextAssembler 负责把占位符替换为实际内容，并保证替换后的总 token ≤ 预算。

## 七、决策记录 (ADR)

### ADR-046: 单轮 input token 预算 8K
- **理由**：128K 窗口虽大，但 input token 成本 ¥0.001/千、output ¥0.002/千，且首 token 延迟随 input 线性增长。8K 是成本/延迟/质量的甜点。
- **替代方案**：32K（成本增 4×，延迟增 2×）、2K（检索质量严重下降）。

### ADR-047: 分层装载 + 动态再分配
- **理由**：不同意图对历史 vs 检索的需求差异大，静态分配浪费预算。
- **实现**：意图分类结果作为分配策略选择信号。

### ADR-048: 本地 Token 估算（jtokkit）而非 API 计数
- **理由**：API 计数需要先发请求，无法在组装阶段做预算控制；本地估算误差 <5% 可接受。
- **校准**：每周采样 100 条真实请求对比估算 vs 实际，偏差 >10% 触发编码表校准。

### ADR-049: 上下文快照入库存档
- **理由**：复盘"为什么答错"必须能看到当时给 LLM 的完整输入。
- **存储**：`agent_context_snapshots` 表，按 session_id+turn_id 唯一，30 天后归档到对象存储。
