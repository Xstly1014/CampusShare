# 查询扩展

> 状态: 草稿  
> 最后更新: 2026-06-30

查询扩展在 `04-intent-understanding/query-rewriting.md` 的同义词扩展基础上，进一步扩大召回。

## 一、扩展手段

### 1.1 同义词扩展（基础）
- 见 `04-intent-understanding/query-rewriting.md` 同义词表。
- OS → 操作系统，扩展为关键词检索的 OR 查询。

### 1.2 相关词扩展（LLM 生成）
- LLM 为 query 生成 3-5 个相关词/近义表达。
- 例：「操作系统期末复习」→ [OS, 操作系统原理, 进程管理, 内存管理, 文件系统, 期末, 期末考试, 复习笔记]
- 这些词用于：
  - 关键词检索扩展（OR）。
  - 向量检索用原 query（向量已含语义，无需扩展词）。

### 1.3 HyDE（假设文档）
- 见 `04-intent-understanding/query-rewriting.md`。
- 用假设文档做向量检索，对短模糊 query 增益大。

### 1.4 多查询生成（Multi-Query，进阶）
- LLM 把原 query 改写成 3 个不同角度的子 query，各自检索后并集。
- 例：「求清华 OS 资料」→
  - 「清华大学 操作系统 期末 复习笔记」
  - 「清华 操作系统 课程 资料 下载」
  - 「清华 OS 习题 答案」
- 三个子 query 各自向量检索 Top-10，并集后 RRF 融合。
- 代价：3 倍向量检索成本，仅进阶开启。

## 二、扩展词的使用边界

- 关键词检索：用扩展词（OR），提升召回。
- 向量检索：用原 query（或 HyDE doc），**不用扩展词**（向量已含语义，扩展词反而引入噪声）。
- 结构化检索：用扩展词匹配分类名/学校名（如「清北」→ 清华+北大）。

## 三、扩展词质量保障

- LLM 生成扩展词时，prompt 约束「只生成相关词，不生成无关热门词」。
- 扩展词去重、去停用词。
- 高频 query 的扩展词缓存（Redis，TTL 1h）。

## 四、负面扩展控制

- 避免过度扩展导致召回噪声：扩展词数 ≤ 5。
- 扩展词与原 query 语义偏离时（LLM 判断）丢弃。

## 五、决策记录 (ADR)

### ADR-027: 扩展词只用于关键词检索，不用于向量检索
- **理由**：向量已捕捉语义，扩展词是关键词检索的补救；向量检索用扩展词反而稀释。
- **HyDE 例外**：HyDE 是生成文档而非词，对向量检索有益。

### ADR-028: Multi-Query 放进阶
- **理由**：3 倍检索成本，MVP 用单 query + 同义词已够；进阶对难例 query 开 Multi-Query。
