# 13 - 评估与质量保障

> 状态: 草稿
> 最后更新: 2026-06-30
> 适用范围: agent-service 全链路质量评估

## 一、本目录定位

本目录解决一个核心问题:**如何持续证明 agent 在变好而不是变坏**。

Agent 系统与传统后端服务的根本区别在于「质量边界模糊」:
- 传统服务:输入确定 → 输出确定,可用单元测试覆盖。
- Agent 服务:输入确定 → 输出可能多种合理形态,且受 prompt/检索/LLM 温度等多因素影响。

因此本目录不追求「断言式测试」,而是构建一套**多维持续评估体系**,从离线黄金集到在线用户信号,形成闭环。

## 四、核心原则

### 原则 1: 离线保下限,在线拉上限
- 离线黄金集用于防止**明显退化**(回归门禁),保下限。
- 在线 A/B 与用户信号用于发现**优化方向**,拉上限。
- 二者不可互相替代:离线集覆盖不了真实长尾,在线信号反馈周期长。

### 原则 2: 指标分层,避免单点优化
评估从不只看一个数字。最低三层:
1. **检索层**:Recall@K、NDCG@10、Hit Rate
2. **生成层**:事实一致性、引用准确率、完整性
3. **体验层**:首 token 延迟、采纳率、重试率

单层最优 ≠ 整体最优。例如检索 Recall 提升但生成质量下降,整体体验可能变差。

### 原则 3: 评估即代码
- 黄金集、评估脚本、Rubric 提示词全部进 Git。
- 评估流水线纳入 CI/CD,PR 合并前必须跑通。
- 评估结果写入 `agent_eval_results` 表,可追溯、可对比。

### 原则 4: 评估者本身也要校准
LLM-as-Judge 不是绝对真理,需要:
- 与人工标注的一致性校验(Cohen's Kappa ≥ 0.6)。
- 偏见检测(位置偏见、长度偏见、自我偏好)。
- 定期用新样本重新校准。

### 原则 5: 样本分层,拒绝「平均分」
不要只看整体平均分。按以下维度切片:
- 意图(HOW_TO / SEARCH / NAVIGATE / CLARIFY / OUT_OF_SCOPE)
- 难度(简单 / 中等 / 困难)
- 输入长度(短 / 中 / 长)
- 是否触发工具调用
- 是否触发澄清

整体持平但某切片暴跌,仍需拦截。

## 六、决策记录

### ADR-169: 评估体系总纲 - 离线在线双轨制
- **背景**:Agent 质量评估既不能只靠离线集(覆盖不足),也不能只靠线上 A/B(反馈慢、成本高)。
- **决策**:采用离线黄金集回归 + 在线 A/B + 用户满意度信号三轨并行。
- **离线轨**:每次 PR 触发黄金集回归,关键指标退化 >5% 阻断合并。
- **在线轨**:新版本灰度 10% 流量,核心指标无退化则全量,周期 ≤7 天。
- **用户轨**:持续采集 👍👎/采纳率/重试率,周报归因分析。
- **权衡**:三轨维护成本高,但相比「线上事故后回滚」的成本可接受。
- **状态**:采纳
