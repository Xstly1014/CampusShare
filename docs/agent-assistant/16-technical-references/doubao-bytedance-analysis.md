# 字节跳动豆包分析

> 状态: 草稿
> 最后更新: 2026-06-30
> 分析对象: 豆包 (字节跳动 AI 助手) + 字节 A/B 测试体系
> 参考: 字节跳动技术博客、火山引擎文档、公开分享

## 一、产品概述

豆包是字节跳动旗下的 AI 助手,定位为「日常 AI 助手」。字节在 AI 领域的独特优势不是模型本身,而是**工程化能力**:
- 海量用户场景下的 A/B 测试体系。
- 内容安全合规(中国大陆市场)。
- 体验度量与数据驱动优化。

## 二、架构拆解(推测)

### 2.1 整体架构

```
用户输入
    │
    ▼
┌──────────────┐
│  意图理解     │  ← 多模型协同(大模型+小模型)
└──────┬───────┘
       ▼
┌──────────────┐
│  检索增强     │  ← RAG + 知识库
└──────┬───────┘
       ▼
┌──────────────┐
│  生成         │  ← 豆包大模型
└──────┬───────┘
       ▼
┌──────────────┐
│  安全审核     │  ← 多层内容安全
└──────┬───────┘
       ▼
┌──────────────┐
│  A/B 分流     │  ← 实验分流
└──────────────┘
```

### 2.2 A/B 测试体系(核心借鉴点)

字节的 A/B 测试文化是其产品成功的基石:

**特点**:
- **一切皆 A/B**:任何产品改动(文案/布局/算法)都需 A/B 验证。
- **快速迭代**:单实验周期 3-7 天,并行多个实验。
- **统计严谨**:样本量计算、显著性检验、切片分析。
- **平台化**:自研 A/B 测试平台,支持正交分层(多实验并行)。

**正交分层(Mutually Exclusive Layers)**:
```
层 1 (UI 实验)     层 2 (算法实验)    层 3 (文案实验)
┌────────────┐    ┌────────────┐    ┌────────────┐
│ A 50%      │    │ C 50%      │    │ E 50%      │
│ B 50%      │    │ D 50%      │    │ F 50%      │
└────────────┘    └────────────┘    └────────────┘
     ↓                  ↓                  ↓
  用户哈希到层1      独立哈希到层2       独立哈希到层3

同一用户在每层独立分桶,实验间不互相干扰
```

### 2.3 体验度量

字节的体验度量体系:

**HEART 框架**(Google 提出,字节采纳):
- **H**appiness:满意度(CSAT/NPS)。
- **E**ngagement:参与度(日活/时长/频次)。
- **A**doption:采纳率(新功能使用率)。
- **R**etention:留存率(次日/7日/30日)。
- **T**ask Success:任务完成率。

**GSM 模型**(Goal-Signal-Metric):
- Goal(目标) → Signal(信号) → Metric(指标)。
- 例如:Goal「回答有用」→ Signal「用户采纳」→ Metric「采纳率」。

### 2.4 内容安全(核心借鉴点)

字节面向中国大陆市场,内容安全是硬性要求:

**多层审核**:
```
输入 → 敏感词库 → 语义分类 → 政治敏感模型 → 风险评估 → 放行/拦截
```

**特点**:
- **实时 + 离线**:实时拦截明显违规,离线复审灰色内容。
- **分级处理**:高危立即拦截,中危人工审核,低危记录。
- **词库热更新**:运营实时增删敏感词,无需发版。
- **多模态**:文本/图片/音频都需审核(豆包支持多模态)。

## 三、可借鉴点

### 3.1 A/B 测试框架(✅ 已借鉴)
→ 见 [13-evaluation/ab-testing-framework.md](../13-evaluation/ab-testing-framework.md)
- 核心指标(采纳率)+ 护栏指标(延迟/错误率)+ 辅助指标。
- 样本量计算、统计显著性检验。
- MVP 阶段单实验互斥,进阶阶段考虑正交分层。

### 3.2 HEART + GSM 体验度量(✅ 已借鉴)
→ 见 [13-evaluation/user-satisfaction-metrics.md](../13-evaluation/user-satisfaction-metrics.md)
- 显式信号(👍👎/CSAT)+ 隐式信号(采纳/重试/中断/回访)。
- 「采纳率」作为核心指标,正是 GSM 模型的应用。

### 3.3 内容安全多层审核(✅ 已借鉴)
→ 见 [14-safety-and-guardrails/content-safety.md](../14-safety-and-guardrails/content-safety.md)
- 三层检测:敏感词库(AC自动机)→ 规则引擎 → LLM 语义分类。
- 词库热更新(管理后台 + Redis Pub/Sub)。
- 灰度内容进人工审核队列。

### 3.4 分级安全处理(✅ 已借鉴)
→ 见 [14-safety-and-guardrails/audit-and-compliance.md](../14-safety-and-guardrails/audit-and-compliance.md)
- P0-P3 安全事件分级。
- 高危立即拦截,中危人工审核。

### 3.5 正交分层 A/B(进阶借鉴)
- MVP 不采用(单实验互斥),但架构预留扩展。
- 进阶阶段实现正交分层,支持多实验并行。

## 四、不适合借鉴点

### 4.1 多模态安全审核
- 豆包支持图片/语音审核,CampusShare MVP 纯文本。
- 多模态审核成本高,无需求。

### 4.2 海量用户 A/B 平台
- 字节自研 A/B 平台支撑亿级用户、数千实验。
- CampusShare 千级用户,MySQL + 简单分桶足够。

### 4.3 实时风控引擎
- 字节有实时风控(反作弊/反爬),CampusShare 规模不需要。
- 用简单的限流 + 行为检测替代。

## 五、关键启发

字节的核心理念:**数据驱动决策,直觉需要验证**。

这对 CampusShare 的启示:
- 不要「我觉得这个 prompt 更好」,要 A/B 验证。
- 不要「用户应该满意」,要采集满意度信号。
- 不要「内容应该安全」,要有多层审核 + 持续监控。

## 六、决策记录

### ADR-190: 字节豆包技术借鉴 - A/B 测试 + 体验度量 + 内容安全
- **背景**:字节在工程化(AB测试/体验度量/内容安全)上业界领先,适合中国大陆场景。
- **决策**:
  - 借鉴:A/B 测试框架(核心+护栏+辅助指标)、HEART+GSM 体验度量、内容安全三层检测、分级安全处理。
  - 不借鉴:多模态审核、海量用户 AB平台、实时风控引擎。
  - 进阶:正交分层 AB(多实验并行)。
- **理由**:字节的工程化方法论可复用,但规模需适配;内容安全是合规硬性要求。
- **状态**:采纳
