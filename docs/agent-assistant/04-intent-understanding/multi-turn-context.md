# 多轮上下文追踪

> 状态: 草稿  
> 最后更新: 2026-06-30

## 一、上下文需要记什么

| 内容 | 用途 | 存储位置 |
|------|------|----------|
| 最近 N 轮对话原文 | 指代消解、追问 | Redis（短期） |
| 上轮检索结果(帖子/知识片段) | CLARIFY 引用 | Redis |
| 上轮意图 | refine/followup 路由 | Redis |
| 用户画像（学校、常问分类） | 个性化 | MySQL（长期） |
| 会话级实体(提到的学校/分类) | 槽位继承 | Redis |

## 二、短期记忆窗口

- **窗口大小**：最近 10 轮（user+assistant 交替）。
- **超窗策略**：旧的轮次压缩为摘要（见 `09-context-engineering/context-compression.md`）。
- **存储结构**（Redis Hash）：
```
key: agent:session:{sessionId}:memory
fields:
  turns: JSON array of {role, content, intent, refs, ts}
  entities: {school, category, post_type}
  summary: "用户在找清华操作系统资料，已推荐3个帖..."
```
TTL：2 小时（用户离开后清理）。

## 三、指代消解触发

当 query 命中以下模式，强制走 CLARIFY/coreference：
- 指代词：「那个」「它」「上面那个」「刚才那个」「第几个」
- 限定词：「有下载的」「带图的」「最热的」（指上轮结果中筛选）

消解方法：
1. 取上轮 `refs`（帖子列表）。
2. 用 LLM 把当前 query 改写为「在上轮结果 [A,B,C] 中筛选 [条件]」。
3. 不重新检索，直接筛选或基于上轮结果生成。

## 四、意图修正（refine）

用户说「我说的是清华不是北大」「不对，我要讨论帖」：
- 检测修正信号词：「不对」「我说的是」「不是xxx是xxx」。
- 从上轮意图 + 槽位出发，应用修正，重走管线。
- 修正后槽位替换原槽位（不是叠加）。

## 五、主题切换检测

用户在同一会话里突然换主题：「顺便问下怎么关通知」：
- 检测「顺便」「对了」「另外」「问下」等转折词 + 新意图与上轮不同。
- 视为新独立问题，上下文不继承（避免污染）。
- 但保留会话 ID（用户视角是连续的）。

## 六、长期记忆（进阶）

- 用户画像表 `agent_user_profile`：
  - `school`：高频提到的学校（用于默认过滤）。
  - `interested_categories`：常问分类。
  - `preferred_post_type`：偏好资源/讨论。
- 每 N 轮异步更新画像，不影响主流程延迟。
- 用途：SEARCH 时默认带上用户学校；推荐时优先偏好分类。

## 七、决策记录 (ADR)

### ADR-014: 短期记忆用 Redis 而非 MySQL
- **理由**：高频读写、TTL 自动过期、低延迟。
- **持久化**：重要会话可异步落 MySQL（用户主动收藏会话时）。

### ADR-015: 指代消解强制 CLARIFY
- **理由**：指代词出现时若误走 SEARCH 重新检索，会丢失「上轮结果」语境，答非所问。
- **代价**：偶尔过度 CLARIFY（用户只是顺口说「那个」）→ 用 LLM 判断是否真指代。
