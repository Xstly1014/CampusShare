## 一、本目录定位

前 16 个目录定义了「Agent 应该长什么样」,本目录回答「**怎么一步步建起来**」。

Agent 工程庞大,不能一次性全建。本路线图将开发分为三个阶段,每阶段有明确交付物和验收标准,确保增量可见、风险可控。

## 四、优先级原则

### 原则 1: 先可用,再优化
- MVP 不追求完美,追求「能用」。
- 检索用简单混合(向量+关键词),不加 reranker。
- 反思不触发,单次 LLM 调用。
- 安全只做基础(输入护栏 + 关键词过滤)。

### 原则 2: 先核心,再外围
- 核心链路(意图→检索→生成)优先于辅助功能(记忆/评估/A/B)。
- 前端聊天 UI 优先于历史会话抽屉。
- 基础监控(QPS/延迟/错误率)优先于高级看板(成本/安全)。

### 原则 3: 先离线,再在线
- 评估先离线黄金集,再在线 A/B。
- 安全先离线敏感词库,再在线 LLM 分类。
- 监控先日志排查,再实时告警。

### 原则 4: 每阶段可独立交付
- MVP 结束即可上线(功能有限但可用)。
- 进阶阶段逐步增强,每功能独立上线。
- 不允许「半成品」跨阶段(如 MVP 就上反思但没评估)。

## 五、技术债务管理

每阶段会产生技术债务(为了速度做的简化),需记录并在后续阶段偿还:

| 债务 | 产生阶段 | 偿还阶段 | 风险 |
|------|----------|----------|------|
| 无 reranker | MVP | 进阶 | 检索质量一般 |
| 无反思 | MVP | 进阶 | 复杂查询质量差 |
| 无上下文压缩 | MVP | 进阶 | 长对话 token 爆炸 |
| 无长期记忆 | MVP | 进阶 | 用户偏好不记忆 |
| 简单限流 | MVP | 进阶 | 可能被滥用 |
| 关键词安全 | MVP | 进阶 | 可能漏检 |
| 单实例 | MVP | 进阶 | 无高可用 |
| 无 A/B | 进阶 | 远期 | 优化无验证 |

## 七、决策记录

### ADR-193: 三阶段渐进式开发路线
- **背景**:Agent 工程庞大,不能一次性全建,需分阶段交付。
- **决策**:MVP(4-6周)→ 进阶(4-6周)→ 远期(持续),每阶段独立可交付。
- **原则**:先可用再优化、先核心再外围、先离线再在线、每阶段独立交付。
- **状态**:采纳
