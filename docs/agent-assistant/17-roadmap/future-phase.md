# 远期阶段规划

> 状态: 草稿
> 最后更新: 2026-06-30
> 阶段: Phase 3 远期
> 周期: 持续(无固定截止)
> 前置: 进阶阶段已完成

## 一、远期目标

**一句话**:在进阶阶段基础上,探索前沿方向,持续优化用户体验和系统能力。

远期阶段不是一次性工程,而是**持续迭代**。以下方向按优先级排列,可根据实际数据反馈调整顺序。

## 二、优化方向

### 2.1 多 Agent 编排(P1)

**目标**:复杂查询由多 Agent 协作完成,提升质量。

**设计**(参考 16-autogen-analysis):
```
用户复杂查询
    │
    ▼
┌──────────┐
│ Router   │  意图分类 + 复杂度评估
│ Agent    │
└────┬─────┘
     │ 简单 → 单 Agent (现有)
     │ 复杂 ↓
┌────▼─────┐
│ Planner  │  任务分解
│ Agent    │
└────┬─────┘
     │
     ├──→ Search Agent (搜索专家)
     ├──→ Help Agent (教程专家)
     └──→ Analyze Agent (分析专家)
          │
          ▼
     ┌────────┐
     │ Critic │  审查综合
     │ Agent  │
     └────────┘
```

**适用场景**:
- 「帮我规划考研复习计划」(需搜索资料 + 整理建议)
- 「这个专业的学习路径是什么」(需多维度搜索 + 综合)

**成本控制**:
- 多 Agent 仅对复杂查询触发(预估 ≤10% 流量)。
- 单次多 Agent 对话成本上限 ¥0.20。
- 超预算降级为单 Agent。

### 2.2 正交分层 A/B(P1)

**目标**:支持多实验并行,提升实验吞吐量。

**设计**(参考 16-doubao-bytedance-analysis):
```
层 1 (检索实验)    层 2 (prompt实验)   层 3 (UI实验)
┌────────────┐    ┌────────────┐    ┌────────────┐
│ A 50%      │    │ C 50%      │    │ E 50%      │
│ B 50%      │    │ D 50%      │    │ F 50%      │
└────────────┘    └────────────┘    └────────────┘
```

- 用户在每层独立哈希分桶。
- 实验间正交,不互相干扰。
- 同时支持 3-5 个实验并行。

### 2.3 Milvus 向量库(P2)

**目标**:替换 PG-Vector,支持更大规模向量检索。

**动机**:
- PG-Vector 在 100 万向量以下性能可接受。
- 帖子量增长到 100 万+ 时,HNSW 检索延迟可能超 SLO。
- Milvus 是专用向量库,支持分布式、十亿级向量。

**迁移计划**:
- 帖子量 < 50 万:继续用 PG-Vector。
- 帖子量 50-100 万:评估 Milvus,做性能对比。
- 帖子量 > 100 万:迁移到 Milvus。

**注意**:Milvus 引入额外运维成本(分布式部署),需评估收益。

### 2.4 Fine-tune 知识库专属模型(P2)

**目标**:用 CampusShare 知识库 fine-tune 一个专属 embedding 或 LLM。

**方向 1:Fine-tune Embedding**
- 用平台帖子/知识文档 fine-tune BGE-M3。
- 提升平台专属术语(如「收纳袋」「创作者认证」)的检索准确率。
- 成本:中等(LoRA fine-tune)。

**方向 2:Fine-tune LLM**
- 用平台高频问答对 fine-tune DeepSeek-V3(若开源支持)。
- 提升回答风格一致性。
- 成本:高(需 GPU 集群)。

**决策依据**:进阶阶段满意度数据。如果通用模型满意度已达 80%+,fine-tune 收益有限。

### 2.5 多模态理解(P3)

**目标**:用户可发送图片,agent 理解图片内容。

**场景**:
- 用户发截图问「这个按钮在哪」→ agent 识别 UI 截图。
- 用户发题目图片问「这道题怎么做」→ agent OCR + 解答。

**技术**:
- 多模态 LLM(GPT-4V / Qwen-VL / 豆包视觉版)。
- 图片上传到 user-service,URL 传给 agent。

**优先级低**:成本高,场景有限,远期考虑。

### 2.6 主动推荐(P3)

**目标**:agent 主动推荐相关内容,而非被动回答。

**场景**:
- 用户浏览帖子时,侧边栏推荐「问问助手:相关学习资料」。
- 用户长时间未使用,推送「你有 3 条新通知,需要助手总结吗?」。

**设计**:
- 基于用户行为(浏览/收藏/搜索)生成推荐 query。
- agent 离线生成推荐回答,缓存。
- 用户点击时直接展示(无延迟)。

### 2.7 知识图谱(P3)

**目标**:构建平台知识图谱,提升推理能力。

**场景**:
- 「线性代数需要什么先修知识」→ 知识图谱推理课程依赖关系。
- 「考研和就业怎么选」→ 图谱关联相关信息。

**技术**:
- Neo4j 图数据库。
- 实体抽取(课程/学校/专业/概念)。
- 关系建模(先修/相关/推荐)。

**优先级最低**:构建成本高,场景有限,远期探索。

### 2.8 高可用部署(P2)

**目标**:多实例 + 负载均衡,消除单点故障。

**设计**:
- agent-service 部署 2+ 实例。
- 网关层负载均衡(已有 Spring Cloud Gateway)。
- Redis 会话锁确保单用户单会话(已有)。
- ShedLock 确保定时任务单实例执行(已有)。

**触发条件**:日活 > 1000 或可用性要求 > 99.5%。

## 三、持续优化方向

除了上述大方向,以下需**持续**优化:

### 3.1 Prompt 持续调优
- 基于满意度数据,每周优化 prompt。
- A/B 验证每次改动。
- Prompt 版本管理(见 08-prompt-versioning.md)。

### 3.2 知识库持续扩充
- 根据用户提问(尤其是 👎 的),补充知识库。
- 每月新增 10-20 篇知识文档。
- 过时文档及时更新。

### 3.3 黄金集持续维护
- 每月补充 10-20 条新样本。
- 季度轮换 20% 防过拟合。
- bad case 驱动补充。

### 3.4 成本持续优化
- 关注模型价格变动(DeepSeek 可能降价)。
- 评估新模型(如 DeepSeek-V4 发布)。
- 优化 token 使用(更短的 prompt、更精准的检索)。

### 3.5 安全持续加固
- 敏感词库持续更新(新词/变体)。
- 红队测试每月新增样本。
- 安全事件复盘改进。

## 四、远期里程碑(弹性)

| 里程碑 | 预估时间 | 依赖 |
|--------|----------|------|
| 多 Agent 编排上线 | 进阶后 2-3 月 | 满意度数据支撑 |
| 正交分层 A/B | 进阶后 1 月 | A/B 框架已就位 |
| Milvus 迁移 | 帖子量达 50 万 | 数据量驱动 |
| Fine-tune 评估 | 进阶后 3 月 | 满意度达瓶颈 |
| 多模态 | 进阶后 6 月 | 需求验证 |
| 高可用部署 | 日活 > 1000 | 流量驱动 |

**注意**:以上时间为弹性预估,根据实际进展调整。远期阶段不设硬性截止日期。

## 五、决策记录

### ADR-196: 远期阶段 - 持续迭代, 数据驱动优先级
- **背景**:远期方向众多,需按优先级和数据反馈安排。
- **决策**:
  - P1:多 Agent 编排、正交分层 A/B。
  - P2:Milvus 向量库、Fine-tune 评估、高可用部署。
  - P3:多模态、主动推荐、知识图谱。
  - 持续:Prompt 调优、知识库扩充、黄金集维护、成本优化、安全加固。
- **原则**:数据驱动优先级,不盲目追新;持续优化优先于新功能。
- **状态**:采纳
