# 成本与延迟分析

> 状态: 草稿  
> 最后更新: 2026-06-30

## 一、单次问答成本分解

以「求清华操作系统期末复习资料」为例：

| 步骤 | 模型 | 输入 tok | 输出 tok | 成本(¥) |
|------|------|----------|----------|---------|
| 意图分类 | V3 | 400 | 30 | 0.00046 |
| 查询改写 | V3 | 500 | 100 | 0.0007 |
| ReAct 第1轮(tool call) | V3 | 1500 | 80 | 0.00166 |
| 最终生成 | V3 | 2500 | 500 | 0.0035 |
| Embedding(query) | BGE-M3 | 50 | - | 0.00005 |
| 重排(进阶) | bge-reranker | - | - | 0.0001 |
| **合计** | | | | **≈ 0.0065** |

远低于 ¥0.03 目标。即使多轮 ReAct 多调一次工具，仍 < ¥0.01。

## 二、延迟分解

| 步骤 | 延迟 |
|------|------|
| 意图分类（非流式） | 400-700ms |
| 查询改写（非流式） | 500-900ms |
| Embedding | 100-200ms |
| 向量检索 + FULLTEXT + 过滤 | 100-300ms |
| 重排（进阶） | 150-300ms |
| ReAct tool call 决策 | 600-1000ms |
| 最终生成首 token | 600-1200ms |
| 最终生成完整(500 tok) | 3-5s |

**首 token P95**：意图分类+改写+检索+tool决策+生成首token ≈ 2-3s。  
> ⚠️ 超过 1.5s 目标。优化见下。

## 三、延迟优化策略

### 3.1 并行化
- 意图分类与查询改写**合并为一次调用**（一个 prompt 同时输出 intent + rewritten query），省 500ms。
- Embedding 与意图分类**并行**（两者输入都来自原 query）。

### 3.2 短路
- 简单 HOW_TO（命中高频 FAQ）跳过 ReAct，直接知识库检索 + 生成。
- CLARIFY 指代类直接从上下文取，不检索。

### 3.3 流式
- 最终生成用流式，首 token 即开始显示，体感延迟降到 1.5s 内。

### 3.4 缓存
- 高频 query 的意图/改写结果缓存（Redis，TTL 1h）。
- Embedding 缓存（query 哈希 → 向量）。

优化后首 token P95 预估 ≤ 1.5s 达标。

## 四、月度成本预估

| 日均问答 | 月成本(¥) | 备注 |
|----------|-----------|------|
| 1,000 | 200 | 当前规模 |
| 5,000 | 1,000 | 增长期 |

Embedding 增量同步帖子：2 万帖全量向量化 ≈ 200 万 tok ≈ ¥2（一次性）。增量每日极少。

## 五、降级成本控制

当主模型限流，切豆包（价格相近，略高）。当所有云端不可用，进入「只读检索模式」：仅返回帖子卡 + 帮助文档片段，不生成自然语言答案，成本 0。
