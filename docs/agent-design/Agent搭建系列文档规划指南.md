# CampusShare Agent 搭建系列文档规划指南

> **本文档用途**：这是 Agent 搭建系列文档的"总纲"。后续 AI 读本文档后，应能知道：
> 1. Agent 搭建一共有哪些方向（完整列表，不遗漏）
> 2. 这些方向该按什么顺序写（顺序规划方法论 + 最终顺序）
> 3. 每个方向写文档时要覆盖什么内容
> 4. 文档写作规范（格式、ADR、范围声明等）
> 5. 当前进度（哪些已完成、下一个该写哪个）
>
> **何时读本文档**：每次开始写新的方向文档前，必须先读本文档，确认方向列表完整、顺序正确、写作规范一致。

---

## 一、为什么需要这份规划指南

### 1.1 踩过的坑

在搭建 Agent 系列文档的过程中，出现过几个严重问题：

1. **方向遗漏**：最初只列了 7 个方向，被指出漏了 MCP；补到 10 个，又被指出漏了知识库管理、代码执行沙箱、多 Agent 协作、LLM 网关、缓存层、限流配额、可观测性、评估体系等 11 个方向。**根因**：站在"当前业务需求"角度精简，而不是站在"大厂完整 Agent 系统"角度系统梳理。

2. **顺序错误**：因为方向不全，顺序自然不对。比如 MCP 应在工具调用之后、对话编排之前，但最初把对话编排放第 6（MCP 都没列入）。

3. **主题混淆**：早期一份文档把 Prompt 工程、上下文工程、对话编排、Agent 范式四个层次混在一起，导致读者迷惑。**根因**：没有先规划方向边界，就开始写。

4. **ADR 未解释**：文档里反复引用 ADR 编号却从不解释 ADR 是什么。

### 1.2 本文档的定位

本文档是"元文档"——不是某个方向的设计方案，而是**所有方向文档的总规划**。它解决三个问题：

- **完整性**：确保方向列表不遗漏（站在大厂角度系统梳理）
- **顺序性**：确保写作顺序符合依赖关系（后续方向能引用前置方向）
- **一致性**：确保所有方向文档的格式、规范、深度一致

## 二、Agent 搭建的完整方向全景

### 2.1 梳理方法

站在大厂角度系统梳理，需要从两个维度交叉验证：

**维度一：系统层次**（从底层基础设施到上层应用）
- 一个生产级 Agent 系统从下到上有：模型层 → Prompt 层 → 知识层 → 理解层 → 记忆层 → 行动层 → 推理层 → 安全层 → 工程层 → 观测层

**维度二：团队职能**（大厂 Agent 团队的角色分工）

- **扩展方向（8 个）**：大厂会建，CampusShare 根据业务需求决定是否独立成文档，或合并到相关方向。

扩展方向的合并建议：

- 18（限流配额与成本控制）→ 可合并到 16（LLM 网关与多模型路由）
- 21（A/B 测试与灰度发布）→ 可合并到 20（评估体系）

## 三、顺序规划方法论

### 3.1 两种规划方法对比

写 21 个方向的文档，顺序怎么定？有两种方法：

#### 方法 A：按层次规划（先定层次顺序，层次内再排方向）

**思路**：先确定 8 大层次的顺序（A→B→C→D→E→F→G→H），层次内再按依赖排方向。

**优点**：
- 层次清晰，先大后小，不会跨层乱跳
- 读者容易理解（按层次递进）
- 适合方向多、层次分明的场景

**缺点**：
- 可能机械地按层次走，忽略跨层依赖
- 比如"知识库管理"（B 层）是"RAG 检索"（B 层）的前置，但"安全护栏"（F 层）其实是横切关注点，不该排在所有方向之后

#### 方法 B：全局规划（不分层次，直接按依赖关系排所有方向）

**思路**：忽略层次，把 21 个方向铺平，纯粹按依赖关系做拓扑排序。

**优点**：
- 严格按依赖关系，每一步都建立在前一步上
- 不会出现"前置方向没写就写后续方向"的问题

**缺点**：
- 可能跨层跳跃，看起来乱
- 21 个方向铺平，依赖关系复杂，拓扑排序结果可读性差

### 3.2 本项目采用的混合方法：依赖驱动 + 层次组织

**结论：采用"依赖驱动的拓扑排序 + 层次组织展示"的混合方法。**

具体规则：

1. **顺序由依赖关系决定**（拓扑排序），不由层次决定
2. **展示按层次分组**，但组内顺序遵循拓扑排序
3. **横切关注点放最后**：安全层（F）、工程基础设施层（G）、观测与评估层（H）贯穿所有层，独立系统化建设，放在核心能力（A-E）之后
4. **允许跨层依赖**：比如"知识库管理"（B 层）是"RAG 检索"（B 层）的前置；"对话编排"（E 层）依赖"工具调用"（D 层）

**为什么用混合方法**：
- 纯按层次：会机械地按 A→B→C 走，但有些跨层依赖会被忽略
- 纯按依赖：21 个方向铺平太乱，可读性差
- 混合：依赖关系保证"顺序正确"，层次组织保证"可读性好"

## 四、各方向写作要点

每个方向的文档应覆盖以下内容（章节结构统一）：

### 4.1 统一章节结构

```
封面（标题 + 副标题 + 版本日期）
本文档范围声明（覆盖什么/不覆盖什么/ADR 解释/与其他文档的关系）
目录
一、场景：为什么需要这个方向（业务背景 + 不用会怎样 + 带来什么 + CampusShare 场景）
二、方案：业界设计模式（技术方案对比 + 大厂案例 + 选型决策 + ADR 汇总）
三、流程：如何搭建（前置条件 + 核心设计 + 整体架构 + 关键决策 ADR）
四、核心代码（文件架构 + 核心类 + 接口改造 + 数据库 Schema）
五、目标：实现效果（功能/性能/质量/成本目标）
六、测试评估与验收（评估指标 + 黄金测试集 + CI-CD + LLM-as-Judge + 测试用例 + A-B + 验收 + 监控）
七、总结与边界声明（核心总结 + 与其他文档关系 + 演进路线）
附录：ADR 摘要
```

### 4.2 各方向的核心内容要点

#### 1. System Prompt 工程
- 六要素结构（角色/能力边界/输出格式/Few-shot/安全护栏/元指令）
- Constitutional AI（5 条规则：角色/能力/指令/隐式指令/信息锁定）
- Prompt 版本管理（SemVer + 灰度发布 + 秒级回滚）
- 注入对抗测试（8 大攻击模式 + 100 条测试用例）
- ADR 前缀：SP

#### 2. 意图识别与路由
- 意图分类体系（HOW_TO/SEARCH/NAVIGATE/CLARIFY/OUT_OF_SCOPE）
- 三层漏斗（规则短路 + LLM 分类 + Embedding 降级）
- 路由策略（不同意图走不同处理链路）
- 意图与工具/上下文的协作
- ADR 前缀：INT

#### 3. 知识库管理（RAG 数据侧）
- 文档摄入流水线（上传 → 解析 → 清洗 → 分块 → Embedding → 入库）
- 分块策略（固定大小/语义分块/递归分块/Markdown 结构分块）
- 增量同步机制（帖子库自动同步、知识库人工维护 + 版本管理）
- 知识库治理（质量评估、过期清理、版本回滚）
- ADR 前缀：KB

#### 4. RAG 检索增强（查询侧）
- 三路召回（向量 + 关键词 BM25 + 结构化过滤）
- RRF 融合（Reciprocal Rank Fusion）
- Cross-encoder 重排
- 知识库与帖子库物理分离
- ADR 前缀：RAG

#### 5. 上下文工程
- L0-L5 六层装载（System Prompt / 用户画像 / 工具 Schema / RAG 结果 / 历史 / 当前输入）
- Token 预算分配（按意图动态分配）
- 三级渐进压缩（Rolling Summary / Slot Freezing / Pin Message）
- 优先级与淘汰策略（P0-P4）
- ADR 前缀：CTX

#### 6. 长期记忆
- 用户画像建模（偏好/历史/行为）
- 记忆持久化（存储结构、检索策略）
- 记忆更新机制（增量更新、遗忘机制）
- 工作记忆与长期记忆的交互
- ADR 前缀：MEM

#### 7. 工具调用（Function Calling）
- OpenAI Function Calling 格式
- 工具定义（注解驱动 + 注册表）
- 工具执行引擎（校验/超时/熔断/脱敏）
- 工具调用循环（最多 5 轮）
- 安全护栏（只读/越权防护/次数上限）
- ADR 前缀：TOOL

#### 8. MCP 协议
- MCP 协议架构（Server/Client/Transport）
- 工具发现与动态注册
- MCP Server 实现（把现有工具暴露为 MCP Server）
- 跨 Agent 工具复用
- 生态接入（社区 MCP Server）
- ADR 前缀：MCP

#### 9. 对话编排
- 范式选型（ReAct / CoT / Plan-and-Execute / Reflexion）
- 多轮流程控制（追问澄清、并行调用、总结收尾）
- 编排与工具调用的协作（ReAct 的 Action = 工具/MCP）
- 编排与上下文工程的协作（多步记忆管理）
- ADR 前缀：DLG

#### 10. 多 Agent 协作（扩展）
- Agent 角色定义（规划者/执行者/审核者）
- 协作模式（流水线/辩论/层级）
- 消息传递与状态同步
- 冲突解决
- ADR 前缀：MAG

#### 11. 安全护栏
- Prompt 注入对抗（直接注入/间接注入）
- 越权调用防护（userId 注入、参数白名单）
- Jailbreak 检测（角色越狱/指令覆盖）
- 安全审计（全链路日志、异常告警）
- ADR 前缀：SEC

#### 12. LLM 网关与多模型路由
- 模型抽象层（统一接口适配 DeepSeek/GPT/Claude）
- 路由策略（按意图/按成本/按延迟/按质量）
- 降级 fallback（主模型挂了切备用）
- 负载均衡（多 API Key 轮询）
- ADR 前缀：GW

#### 13. 可观测性
- 全链路追踪（Trace：用户请求 → 意图 → RAG → 工具 → LLM）
- Metrics 指标（延迟/Token/成本/错误率/工具调用率）
- Logs 日志（结构化日志、agent_turns/tool_calls 表）
- 告警规则（延迟/错误率/成本阈值）
- ADR 前缀：OBS

#### 14. 评估体系
- 评估指标体系（按方向分层：Prompt/RAG/意图/上下文/工具/编排）
- 黄金测试集构建与维护
- CI/CD 集成（PR 触发评估）
- LLM-as-Judge（多维 Rubric）
- 影子评估（线上流量采样对比）
- 回归检测（与上次版本对比）
- BadCase 驱动的数据飞轮（生产失败案例自动采集 → 分诊标注 → 黄金集扩充 → 改进 → 上线 → 再采集，形成闭环）
- 业务感知评测指标（不只看技术指标如 BLEU/F1，更看业务 KPI 如任务完成率/用户负反馈率/转人工率）
- ADR 前缀：EVAL

#### 15. 分层部署与在离线协同
- 组件分层决策（在线：对话/意图/RAG/工具调用；异步：知识库摄入/Embedding 生成/记忆整合/审计；离线：黄金集评估/批量分析/报表）
- 在离线协同链路（事件总线/消息队列/CDC 数据同步/定时任务编排）
- 数据一致性保证（最终一致 vs 强一致场景区分）
- 资源隔离与弹性伸缩（在线优先保障、离线可抢占）
- ADR 前缀：DEPLOY

#### 16. 性能 SLO 工程
- 核心场景 SLO 定义（对话 TTFT < 800ms / P99 < 3s；搜索 P99 < 1.5s；工具调用 P99 < 2s）
- 延迟预算分配（总预算拆分到意图检测/RAG/LLM/工具调用各环节）
- SLO 驱动架构决策（超预算时的降级策略：跳过重排/减少历史轮数/切换小模型）
- SLO 燃烧率告警（基于多窗口多燃烧率算法，避免告警风暴）
- SLO 仪表盘与错误预算管理
- ADR 前缀：SLO

## 五、文档写作规范

### 5.1 格式规范

1. **一份文档只讲一个细小方向**，不混入其他层次的内容
2. **开篇有"本文档范围声明"页**：明确覆盖什么、不覆盖什么、ADR 解释、与其他文档的关系
3. **ADR 用前缀编号**：每个方向有自己的前缀（SP/INT/KB/RAG/CTX/MEM/TOOL/MCP/DLG/MAG/SEC/GW/DEPLOY/OBS/SLO/EVAL）
4. **文末附录列出全部 ADR 摘要**：每条 ADR 包含背景/决策/理由/后果
5. **第七章列出与其他文档的关系和演进路线**

### 5.2 章节结构统一

所有方向文档都遵循"场景 → 方案 → 流程 → 代码 → 目标 → 测试 → 总结"七章结构（见 4.1 节）。

### 5.3 代码规范

- 代码块用灰色背景等宽字体（Consolas）
- Java 文本块 `"""..."""` 在 Python 脚本中用 `'''..."""..."""...'''` 包裹避免冲突
- 代码要基于真实文件架构，不要凭空捏造
- 改造现有代码时，标注"改造前/改造后"

### 5.4 表格规范

- 表头深色背景 + 白色加粗字体
- 列宽合理，避免内容挤压
- 隔行变色（可选）

### 5.5 生成方式

- 用 Python 脚本（python-docx 库）生成 .docx 文件
- 脚本命名：`generate-{方向英文缩写}-doc.py`
- 输出文件命名：`{方向中文名称}模块设计方案.docx`
- 脚本和文档都放在 `docs/agent-design/` 目录

## 七、后续 AI 的工作流程

### 7.1 写新方向文档前的检查清单

每次开始写新的方向文档前，AI 必须执行以下步骤：

1. **读本文档**：确认方向列表完整、顺序正确、写作规范一致
2. **确认当前进度**：查看第六节，确认哪些已完成、下一个该写哪个
3. **确认方向归属**：该方向属于哪个层次？是核心还是扩展？
4. **确认依赖关系**：该方向的前置方向是否都已完成？如果前置方向未完成，先补写前置方向
5. **确认 ADR 前缀**：用第四节 4.2 节的 ADR 前缀，避免冲突
6. **读前置方向文档**：确保新文档能正确引用前置方向的概念和 ADR

### 7.2 写文档时的规范检查

1. **范围声明**：开篇是否有"本文档范围声明"页？
2. **单主题**：是否只讲一个方向，没有混入其他层次？
3. **ADR 解释**：第一次出现 ADR 时是否解释了什么是 ADR？
4. **七章结构**：是否遵循"场景→方案→流程→代码→目标→测试→总结"结构？
5. **代码真实性**：代码是否基于真实文件架构和现有代码？
6. **与其他文档关系**：第七章是否说明了与前置/后续文档的关系？

### 7.3 写完文档后的更新清单

1. **更新本文档第六节**：把新完成的文档加入"已完成"列表
2. **更新前置文档的"演进路线"**：如果新文档是前置文档的后续，在前置文档的第七章演进路线中标注"已完成"
3. **生成 .docx 文件**：运行 Python 脚本生成 Word 文档
4. **验证文件**：确认 .docx 文件大小合理（通常 60-100KB）

### 7.4 顺序规划方法回顾

当需要重新审视顺序时（比如新增方向、调整依赖关系），按以下步骤：

1. **确认方向列表完整**：站在大厂角度，按"系统层次 + 团队职能"两个维度交叉验证
2. **分析依赖关系**：每个方向的前置依赖是什么？被谁依赖？
3. **拓扑排序**：按依赖关系排序，横切关注点放最后
4. **层次组织展示**：按层次分组，但组内顺序遵循拓扑排序
5. **更新本文档**：把新的方向列表和顺序写入本文档

**关键原则**：
- 顺序由依赖关系决定，不由层次决定
- 展示按层次分组，保证可读性
- 横切关注点（安全/网关/观测/评估）贯穿所有层，放最后系统化建设
- 允许跨层依赖（如对话编排依赖工具调用）

## 八、关键认知（避免重复犯错）

### 8.1 方向完整性认知

**错误**：站在"当前业务需求"角度精简方向列表
**正确**：站在"大厂完整 Agent 系统"角度系统梳理，再按业务需求标注核心/扩展

**验证方法**：从"系统层次"和"团队职能"两个维度交叉验证，确保不遗漏。

### 8.2 顺序规划认知

**错误**：按层次机械排序（A→B→C→D→E→F→G→H）
**正确**：按依赖关系拓扑排序，层次只用于组织展示

**关键**：横切关注点（安全/网关/观测/评估）不是"某一层"，而是"贯穿所有层"的独立方向，放最后系统化建设。

### 8.3 主题边界认知

**错误**：一份文档混入多个层次（如把 Prompt 工程、上下文工程、对话编排混在一起）
**正确**：一份文档只讲一个细小方向，开篇用"范围声明"明确边界

**比喻**：想想你是刚进字节跳动的员工，你不可能一个人负责那么多模块，肯定是针对一小个内容深入。

### 8.4 MCP 与工具调用的关系认知

**错误**：把 MCP 当成工具调用的"一种方案"
**正确**：MCP 是独立的"协议层/生态层"，工具调用是"机制层"

- 工具调用（Function Calling）解决"LLM 怎么调工具"
- MCP 解决"工具从哪来、怎么复用、怎么跨 Agent 共享"
- 两者不能互相替代，都是独立方向

### 8.5 Agent 开发边界认知（什么不属于 Agent 开发）

以下方向曾被提出但**不纳入**本系列文档，因为它们属于模型团队/MLOps 团队职责，不属于 Agent 开发者范围：

| 被排除方向 | 排除理由 | 属于谁 |
|---|---|---|
| 范式选型（LLM/VLM/垂类模型选型） | 模型范式选型是模型团队的设计时决策；Agent 编排范式已由对话编排（#12）覆盖；运行时多模型路由已由 LLM 网关（#16）覆盖 | 模型架构团队 |
| SFT / 偏好对齐（RLHF/DPO/PPO） | 需要模型权重和训练基础设施；CampusShare 使用 DeepSeek API 不涉及训练；Agent 开发者通过 RAG/Prompt/知识库注入知识 | 模型训练团队 |
| 推理加速与压缩（量化/蒸馏/投机解码） | 量化/蒸馏/投机解码需要模型权重，API-based Agent 无法操作；语义缓存已在缓存层（#17）覆盖 | 模型推理团队 |

**判断标准**：如果一个方向需要"模型权重"或"训练基础设施"，它就不属于 Agent 开发范围。Agent 开发者的知识注入路径是 RAG + Prompt + 知识库 + 长期记忆，不是 SFT。
