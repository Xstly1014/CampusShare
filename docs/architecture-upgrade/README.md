## 二、升级背景与原则

### 2.1 当前现状

CampusShare 作为校园资源共享平台，目前处于 MVP 阶段。现有技术选型以"快速实现功能"为首要目标，存在大量"怎么简单怎么来"的草率决策。随着功能复杂度增加和用户量预期增长，当前架构已显现出以下瓶颈：

1. **微服务架构不完整**：名义上是微服务，实际共享数据库，缺少服务注册发现、配置中心、熔断降级等微服务基础设施
2. **数据层过于简单**：单库单表、没有消息队列、没有分布式事务、没有数据一致性方案
3. **工程化程度低**：缺少单元测试、集成测试、CI/CD、代码质量门禁
4. **性能与扩展性不足**：轮询替代 WebSocket、本地文件存储、没有搜索基础设施
5. **安全防护薄弱**：简单 JWT、没有接口限流（代码层面）、没有风控、没有审计日志
6. **可观测性不完善**：Metrics 有了但 Logs 缺失，没有告警通知渠道

### 2.2 升级核心原则

> ⚠️ **Agent 在执行升级时必须严格遵守以下原则**

1. **专业优先，禁止偷懒**：技术选型不能因为"简单"就选，必须基于性能、可维护性、扩展性、社区生态综合评估
2. **渐进式升级，不破坏现有功能**：分阶段实施，每个阶段完成后现有功能必须正常运行
3. **每个升级必须有数据支撑**：升级前后要有性能/可维护性/稳定性的对比数据
4. **中间件引入要谨慎但不畏惧**：该引入的中间件必须引入（如消息队列、注册中心、ES），不要因为"增加复杂度"而回避
5. **文档同步更新**：每完成一个升级阶段，同步更新对应文档、AGENT-WORKFLOW.md、README.md
6. **测试先行**：引入新组件或重构核心模块前，先补充测试用例

### 2.3 升级阶段划分

| 阶段 | 时间 | 核心目标 |
|------|------|----------|
| **Phase 1: 基础设施补全** | 2-3周 | 服务注册发现、配置中心、消息队列、日志聚合、分布式锁、限流熔断 |
| **Phase 2: 数据层升级** | 2-3周 | 数据库垂直拆分、读写分离规划、Elasticsearch搜索、分布式事务、多级缓存 |
| **Phase 3: 实时通信升级** | 1-2周 | WebSocket 替代轮询、消息推送、在线状态 |
| **Phase 4: 工程化体系建设** | 2周 | 单元测试、集成测试、CI/CD、代码质量、API文档自动化 |
| **Phase 5: 性能与稳定性** | 持续 | JVM深度调优、SQL优化、压测、容灾、监控告警完善 |
| **Phase 6: 前端架构升级** | 与后端并行 | 数据缓存层、状态管理优化、组件库建设、虚拟滚动、PWA |

## 三、升级优先级定义

| 级别 | 含义 | 示例 |
|------|------|------|
| **P0 - 必须立即做** | 架构缺陷，不做会导致严重问题或无法扩展 | 引入消息队列解耦、服务注册发现 |
| **P1 - 近期必须做** | 明显的技术债务，影响开发效率或稳定性 | 引入分布式锁、单元测试框架 |
| **P2 - 计划中** | 提升体验或性能，在功能稳定后实施 | WebSocket、Elasticsearch、读写分离 |
| **P3 - 长期规划** | 架构演进方向，用户量达到一定规模后实施 | 分库分表、Service Mesh、容器编排 |

## 四、给 Agent 的特别说明

1. **不要怕引入复杂中间件**：Nacos、RocketMQ、Elasticsearch、Sentinel、Seata 这些都是微服务的标准组件，该用就用
2. **每个中间件引入必须有文档**：为什么选它、备选方案是什么、怎么配置、怎么监控、故障怎么排查
3. **优先保证可观测性**：引入任何新组件，必须同时配置监控指标、日志、链路追踪
4. **Docker Compose 必须同步更新**：新组件必须加入 docker-compose.yml，配置好健康检查、数据卷、网络
5. **AGENT-WORKFLOW.md 必须同步更新**：新组件的配置、端口、调试命令、踩坑记录必须追加到 AGENT-WORKFLOW.md 对应章节
6. **升级过程必须记录到 optimization-logs/**：每个升级完成后，记录「问题→方案→实施→验证」全过程，作为简历亮点素材
