# 优化计划 08：评论列表 + 创建

> **优先级：P1** | **服务：post-service(8082) + user-service(8081)** | **接口数：2**

---

## 一、业务背景

### 1.1 业务背景类型：被动场景型（系统/运行时驱动）

**为什么现在做**：评论列表是帖子详情页必调接口，需组装楼中楼树形结构 + 作者信息，N+1 风险高。评论创建涉及跨服务通知（COMMENT/REPLY 类型）+ 帖子 comment_count 更新，是复合写操作。

**如果不做会怎样**：热门帖子评论数百条时树形组装 + 逐条查作者导致延迟劣化；评论创建同步发通知阻塞。

**做成之后意味着什么**：评论列表在百条评论下 P95 < 300ms；评论创建 < 200ms（通知异步化）。

### 1.2 涉及接口

| # | 接口 | 方法 | 路径 | 当前实现要点 |
|---|------|------|------|-------------|
| 1 | 评论列表 | GET | `/posts/{postId}/comments` | 查评论 + 树形组装(楼中楼) + 作者信息(Feign) + 点赞状态 |
| 2 | 评论创建 | POST | `/posts/{postId}/comments` | 写 comments + 更新 post.comment_count + Feign 发 COMMENT/REPLY 通知 |

## 二、当前实现分析

> ⚠️ 优化前需 `Read` CommentController + CommentServiceImpl 确认

### 2.1 评论列表疑似瓶颈

| 编号 | 疑点 | 风险等级 | 说明 |
|------|------|----------|------|
| CL1 | **作者信息 N+1** | 高 | 每条评论查作者 Feign（若未批量） |
| CL2 | **树形组装性能** | 中 | 楼中楼 parent_id 递归/多层组装 |
| CL3 | **评论点赞状态 N+1** | 中 | 每条评论查当前用户是否已赞 |
| CL4 | **无分页** | 中 | 评论是否一次性返回全部？需确认（热门帖数百评论） |
| CL5 | **无缓存** | 低 | 评论列表易变，缓存价值低但可考虑短TTL |

### 2.2 评论创建疑似瓶颈

| 编号 | 疑点 | 风险等级 | 说明 |
|------|------|----------|------|
| CC1 | **通知发送同步阻塞** | 高 | Feign 发 COMMENT/REPLY 通知同步等待 user-service |
| CC2 | **comment_count 行锁** | 中 | UPDATE post SET comment_count=comment_count+1 |
| CC3 | **事务边界** | 中 | 事务是否包 Feign 调用（长事务） |

## 三、优化方案

### 3.1 评论列表优化

**优化1：批量查询作者信息**
- 收集所有评论 userId → 批量 Feign `getBatchUserInfo`
- 复用帖子列表的批量查询模式

**优化2：评论点赞状态批量查**
- 一次查当前用户对所有评论的点赞记录（`comment_likes WHERE user_id=? AND comment_id IN (...)`）

**优化3：分页**
- 根评论分页（如每页20条根评论），子评论随根评论返回
- 避免一次性返回数百评论

**优化4：树形组装优化**
- 内存中用 Map<parentId, List<Comment>> 组装，避免递归查DB

### 3.2 评论创建优化

**优化1：通知异步化**
- Feign 通知改 @Async
- 通知失败不影响评论创建

**优化2：计数原子化**
- 确认 setSql 原子操作

**优化3：事务边界**
- 事务只包 comments insert + post.comment_count update
- Feign 通知在事务外

### 3.3 方案选型

| 方案 | 核心思路 | 选用？ |
|------|----------|--------|
| **A: 批量查询 + 异步通知** | 复用已验证模式 | ✅ |
| B: 评论列表缓存 | 短TTL(30s) | ⚠️ 第二阶段（易变） |
| C: 评论树缓存 | 按帖子缓存树结构 | ⚠️ 第二阶段 |
