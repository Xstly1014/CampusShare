## 一、业务背景

### 1.1 业务背景类型：被动场景型（系统/运行时驱动）

**为什么现在做**：`/posts/school-counts` 是首页加载必调接口，历史已优化（SQL GROUP BY + Redis 缓存，P95 16.3s→<100ms）。当前需复检确认优化是否仍生效、缓存命中率是否达标、数据量增长后是否需要进一步优化。

**如果不做会怎样**：优化回退或数据量增长后聚合查询再次劣化。

**做成之后意味着什么**：验证历史优化持久性，确认缓存策略在数据量增长后仍有效。

### 1.2 涉及接口

|---|------|------|------|------|

## 二、复检要点

### 2.1 历史优化内容

来自 [insights/performance 接口性能SQL聚合缓存N+1](../../insights/performance/2026-06-29_optimization_接口性能SQL聚合缓存N+1.md)：
- ✅ SQL GROUP BY 替代内存聚合
- ✅ Redis 缓存 `cache:school:post:counts`，TTL 5分钟
- ✅ P95 从 16.3s 降至 < 100ms（缓存命中 < 10ms）

### 2.2 复检项

| 复检项 | 方法 | 达标标准 |
|--------|------|----------|
| 复检1: P95 延迟 | JMeter 压测 | < 100ms（缓存命中）/ < 200ms（缓存未命中） |
| 复检2: 缓存命中率 | Redis `INFO stats` 或 Grafana | > 90% |
| 复检3: SQL 是否仍用 GROUP BY | `Read` 源码确认 | 未回退 |
| 复检4: 数据量增长后性能 | 压测 10万帖子 | 仍达标 |

### 2.3 进一步优化机会

| 优化项 | 方案 | 优先级 |
|--------|------|--------|
| 主动失效 | 发帖时主动更新/失效缓存 | 中 |
| 本地缓存 | Caffeine 二级缓存（应用内） | 低 |
| 计数预计算 | 定时任务预计算（如数据量极大） | 低（当前量级不需要） |
