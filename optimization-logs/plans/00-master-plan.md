## 一、优化目标

### 1.1 业务目标

CampusShare 已完成 MVP，所有核心功能跑通。当前进入**性能优化阶段**，首要任务是降低接口延迟，为后续压测调优、技术方案优化打基础。

### 1.2 技术目标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 核心读接口 P95 | < 200ms | JMeter 压测 |
| 核心写接口 P95 | < 300ms | JMeter 压测 |
| 列表接口 P95 | < 500ms | JMeter 压测 |
| 文件上传接口 | 大文件(50MB) < 5s | curl + 计时 |
| AI对话首token | < 2s | 前端计时 |
| 系统整体 QPS | > 500 req/s | JMeter 压测 |

### 1.3 简历/面试目标

每个核心接口优化后，在 `records/` 产出一条可面试的 STAR 记录，覆盖：
- 业务背景（主动需求型/被动场景型）
- 延迟根因（代码/SQL/架构/JVM/缓存）
- 优化方案（含技术选型对比）
- 量化结果（前后对比数据）

## 三、排期与依赖

### 3.1 建议执行顺序

> 依据：先易后难、先核心后常规、先单服务后跨服务、先代码后架构

```
批次1：P0 高并发写（单服务内，影响面大，方案相对独立）
  ├─ ✅ 帖子点赞/收藏/下载记录 (01) ← 2026-07-03完成
  └─ ✅ 评论点赞 (02) ← 2026-07-03完成

批次2：P0 文件IO
  └─ ✅ 文件上传 (03) ← 2026-07-03完成（生产级加固：并发限流+磁盘保护+原子写）

批次3：P0 复合操作
  └─ 🔜 发帖 (04，依赖03的文件处理经验) ← 下一批次

批次4：P0 读写混合
  └─ 帖子详情 (05)

批次5：P1 读密集列表（N+1模式，可复用批量查询方案）
  ├─ 帖子列表查询族 (06，复检历史优化)
  └─ 学校帖子计数 (07，复检历史优化)

批次6：P1 评论/通知
  ├─ 评论列表+创建 (08)
  └─ 通知查询族 (09)

批次7：P2 常规
  ├─ 私信 (10)
  ├─ 用户/统计聚合 (11)
  ├─ 登录 (12)
  └─ AI对话 (13)
```

### 3.2 依赖关系

| 接口 | 依赖 | 说明 |
|------|------|------|
| 发帖(04) | 文件上传(03) | 发帖含文件上传，复用文件处理优化经验 |
| 帖子列表(06) | 无 | 历史已优化，复检验证 |
| 学校计数(07) | 无 | 历史已优化，复检验证 |
| 评论列表(08) | 帖子详情(05) | 评论列表依赖帖子存在性校验 |
| 通知查询(09) | 点赞(01) | 点赞触发通知，通知查询依赖通知写入 |
| 私信发送(10) | 无 | 独立 |
| 用户主页(11) | 帖子统计 | 跨服务 Feign 聚合 |
