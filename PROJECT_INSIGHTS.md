> **给 Agent / 开发者的指令（请先阅读本节）**
>
> 你不是在写 README，不是在写 API 文档，不是在写变更日志。
> 你的任务只有一个：**为这个项目的简历和面试积累不可替代的亮点素材**。
>
> ---
>
> ## 核心方法论：STAR 法则
>
> 每一个记录文件，本质上都是在讲一个 **STAR 故事**：
>
> | STAR | 含义 | 在本记录体系中的对应部分 |
> |------|------|--------------------------|
> | **S** — Situation | 当时是什么情况、什么约束、什么问题 | 各模块的「背景与问题」 |
> | **T** — Task | 需要你干什么、目标是什么 | 各模块的「目标 / 设计目标」 |
> | **A** — Action | 你做了什么、怎么做的、为什么这样做 | 各模块的「决策 / 方案 / 实现细节」 |
> | **R** — Result | 带来了什么结果、有什么数字 | 各模块的「影响 / 结果 / 前后对比」 |
>
> **每个记录文件都必须覆盖 SATR 四个环节。如果有任何一个环节缺失，这份记录在简历和面试中就毫无价值。**
>
> ---
>
> ## 业务背景的分类与写法（STAR — S 的核心，最关键的一环）
>
> **每一个 STAR 故事的起点都是一个清晰的业务背景。没有业务背景的技术方案是"为做而做"，在简历和面试中毫无说服力。业务背景回答的是"为什么需要做这件事"，它决定了后续所有技术决策的合理性——如果业务背景站不住，后面的方案再精巧也是空中楼阁。**
>
> 业务背景分为两类，每类有不同的触发来源和写作重点。**每条记录必须在 S 环节开头标注业务背景类型**：
>
> ### 类型一：主动需求型（用户 / 业务方驱动）
>
> **特征：** 业务方或用户明确提出需求，技术方案是为了满足这个需求。需求来自"外面"。
>
> **写作要点：**
> - 谁提出的需求？（用户反馈 / 产品规划 / 业务方诉求）
> - 需求的具体内容是什么？（要具体，不要笼统说"优化体验"）
> - 为什么现有方案无法满足？（能力缺失 / 性能不足 / 功能不存在）
> - 如果不做会怎样？（用户流失 / 体验差 / 业务无法开展）
>
> **典型示例：**
> - "用户希望 Agent 助手能维持长上下文，在多轮对话中记住之前的讨论内容" → 触发了上下文管理策略的设计
> - "用户希望 Agent 能自己调用 MCP 实现操作，而不仅仅是回答文字" → 触发了工具调用能力的开发
> - "用户希望 Agent 的回答更精准，减少幻觉和无关内容" → 触发了 RAG 检索增强和 Prompt 优化
>
> **效果数据重点：** 用户满意度提升、功能使用率、任务完成率、回答准确率等**用户体验指标**。
>
> ### 类型二：被动场景型（系统 / 运行时驱动）
>
> **特征：** 不是用户主动要求，而是系统运行或业务发展中暴露出的问题。需求来自"里面"。
>
> **写作要点：**
> - 什么场景下暴露的问题？（高并发 / 数据量增长 / 边界情况 / 外部依赖异常）
> - 问题的症状是什么？（响应慢 / 报错 / 数据不一致 / 资源耗尽）
> - 影响范围多大？（多少用户 / 多少请求 / 多少数据，**必须量化**）
> - 如果不解决会怎样？（系统崩溃 / 数据丢失 / SLA 违约）
>
> **典型示例：**
> - "大量用户同时点赞一条热门帖子，造成数据库行锁竞争和高并发写入压力，P99 延迟飙升至 2s+" → 触发了异步化 + 缓存优化
> - "随着帖子数量增长到 10 万+，全文搜索 P99 延迟超过 3s，用户反馈搜索卡顿" → 触发了 Elasticsearch 迁移
> - "DeepSeek API 突然限流，导致 Agent 服务对所有用户返回 503" → 触发了熔断降级 + 缓存兜底
>
> **效果数据重点：** 性能提升（延迟 / QPS / 吞吐）、资源节省、错误率下降、可用性提升等**系统指标**。
>
> ### 通用要求（两类都必须回答）
>
> 1. **为什么是现在做？**（触发时机和紧迫性——不是"早就该做"，而是具体的事件或数据触发了决策）
> 2. **如果不做会怎样？**（后果和影响——量化损失，不要只说"体验不好"）
> 3. **做成之后意味着什么？**（业务价值 + 技术价值——面试官想听的是"so what"）
>
> **面试核心逻辑：** 面试官最关心的不是"你用了什么技术"，而是"你为什么觉得这个问题值得解决"。
> 一个好的业务背景，能让面试官立刻理解技术方案的必要性，产生"如果是我也会想解决"的共鸣。
> 一个差的业务背景，会让面试官追问"那为什么要做这个？"，你就陷入了被动。
>

> ## 关于图表：图文并茂是强制要求
>
> 纯文字的架构描述 = 废纸。以下类型的记录，**强制要求附带 Mermaid 图**：
>
> | 记录类型 | 强制图表类型 | Mermaid 语法 |
> |----------|-------------|-------------|
> | ADR（架构决策） | 方案对比流程图 / 组件交互图 | `flowchart LR/TB` 或 `graph` |
> | 数据模型设计 | ER 图 | `erDiagram` |
> | 接口与协议设计 | 序列图 | `sequenceDiagram` |
> | 系统整体架构 | 组件图 / C4 容器图 | `flowchart TB` 或 `graph TB` |
> | 数据流 / 管道设计 | 数据流图 | `flowchart LR` |
> | Agent 工作流 | 状态图 + 序列图 | `stateDiagram-v2` + `sequenceDiagram` |
> | 容错与降级 | 决策树 / 流程图 | `flowchart TD` |
> | 部署拓扑 | 部署图 | `flowchart TB`（用 subgraph 分组） |
>
> **图表质量标准：**
> - 每个节点必须有明确的标签（不是缩写）
> - 多条路径的，标注判断条件在边上
> - 数据流图标明数据格式或协议
> - 序列图标注超时、重试等非功能属性
> - 不要画"教科书式"的通用图，要画**你这个项目特定**的图
>

> ## 文件夹结构约定
>
> 阅读本文档后，在项目根目录创建 `insights/` 文件夹，结构如下：
>
> ```
> insights/
> ├── architecture/       ← 架构决策与技术选型（开发阶段重点填充）
> │   ├── 2024-03-15_ADR-001_消息队列选型.md
> │   ├── 2024-03-16_tech-selection_数据库.md
> │   ├── 2024-03-18_data-model_核心实体.md
> │   ├── 2024-03-20_api-design_任务管理模块.md
> │   ├── 2024-03-22_system-overview_整体架构.md    ← 系统全景图（必须）
> │   └── ...
> ├── performance/        ← 性能测试与优化记录（测试/优化阶段填充）
> ├── reliability/        ← 稳定性、容错与故障复盘（上线后填充）
> ├── agent-eval/         ← Agent 评估体系（Agent 项目专用）
> ├── engineering/        ← 工程化质量记录（贯穿全程）
> ├── resume-bullets.md   ← 简历条目提炼（阶段性生成）
> └── README.md           ← 本文件夹的索引（自动生成）
> ```
>
> **文件命名规则：** `YYYY-MM-DD_类型_简短描述.md`
>
> **阶段指引：**
> - 开发阶段 → 重点填充 `architecture/`，兼顾 `engineering/`（测试策略、可观测性设计）
> - 测试/优化阶段 → 重点填充 `performance/`，回补 `architecture/` 中遗漏的决策
> - 上线/运维阶段 → 重点填充 `reliability/`
> - Agent 项目 → 全程同步填充 `agent-eval/`
> - 每个里程碑节点 → 更新 `resume-bullets.md`

## 开发中的触发规则

**每当你做以下事情时，必须创建或更新 insights 记录：**

| 触发事件 | 动作 | 目标文件 |
|----------|------|----------|
| 选择/更换框架或库 | 创建技术选型记录 | `architecture/` |
| 设计/修改数据库 Schema | 创建数据模型记录（含 ER 图） | `architecture/` |
| 设计一个新的 API 或消息协议 | 创建接口设计记录（含序列图） | `architecture/` |
| 面临两个以上的方案选择 | 创建 ADR（含决策流程图） | `architecture/` |
| 跑完一轮压测 | 创建基准测试记录 | `performance/` |
| 做了一次性能优化 | 创建优化记录（含前后对比） | `performance/` |
| 处理了一个容错/降级场景 | 更新容错设计记录 | `reliability/` |
| 线上出现问题 | 创建故障复盘 | `reliability/` |
| Agent 跑了一轮 eval | 创建评估运行记录 | `agent-eval/` |
| 改了 Prompt 策略 | 创建 Prompt 工程记录（含版本树） | `agent-eval/` |
| 完成一个里程碑 | 更新 resume-bullets.md | `insights/` |

## 图表质量自查

创建每个文件前，确认：
1. 是否至少包含一张 Mermaid 图（ADR、数据模型、接口、组件交互、Agent 工作流必须有图）
2. 图是否具体到本项目（不能是通用教科书图）
3. 图是否标注了关键决策点、异常路径、超时、重试
4. 图的节点标签是否清晰（不是缩写，外人能看懂）
5. STAR 四个环节是否都有内容（S 和 R 可以事后回填，但必须预留位置）
6. **S 环节是否标注了业务背景类型**（主动需求型 / 被动场景型），并回答了"为什么现在做""不做会怎样""做成意味着什么"三个问题——这是整个记录的价值根基
