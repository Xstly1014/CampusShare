## [2026-07-04] 下载记录接口根因修复：download_count 计数字段 + 去重机制，彻底解决数据膨胀问题

### 性能优化 + Bug修复

**问题**：`recordDownload` 每次下载都 INSERT 一行到 `post_downloads`，无去重无计数聚合。压测65万次后 `post_downloads` 表再次膨胀，warehouse-stats/my-downloads 接口面临再次劣化风险。之前仅做数据清理治标不治本。

**根因**：
1. `post_downloads` 无唯一约束，同一用户对同一帖子反复下载产生重复行
2. `posts` 表缺少 `download_count` 字段，无法像 `view_count`/`like_count`/`star_count` 那样原子计数
3. `totalDownloadsOfMyPosts` 使用 `COUNT(*) INNER JOIN post_downloads` 查询，大表下慢
4. `PostDownloadMapper` 无并发安全保护（DuplicateKeyException 未处理）

**优化方案**（参照 view_count 模式）：
1. **posts 表新增 `download_count INT DEFAULT 0`**，每次下载原子 `SET download_count = download_count + 1`
2. **post_downloads 添加唯一索引 `uk_user_post(user_id, post_id)`**，同一用户对同一帖子只保留一条记录
3. **recordDownload 去重逻辑**：先查是否已存在记录 → 存在则 UPDATE download_time，不存在则 INSERT → catch DuplicateKeyException 幂等处理
4. **getWarehouseStats**：`totalDownloadsOfMyPosts` 改为读 `SUM(download_count)` FROM posts（直接读计数字段，无需 JOIN post_downloads）
5. **删除慢查询**：移除 `countDownloadsByAuthor`（COUNT(*) INNER JOIN 全表扫描），避免误用
6. **DTO 层补齐**：PostDetailDTO/PostListDTO 添加 downloadCount 字段；PostController/InternalPostController/CategoryController 补齐 setDownloadCount

**压测验证**：656,034 samples, 0% error, avg 13ms, QPS 3644/s（清理前1164/s），唯一索引保证不膨胀

## [2026-07-03] 性能优化：点赞/收藏接口并发错误率 38%→0%，QPS提升2.9倍（180秒热JVM）

### 性能优化

**问题**：JMeter 50并发基线压测发现 `POST /api/posts/{id}/like` 和 `/star` 接口错误率高达37-38%，大量HTTP 500。

**根因**：
1. `toggleLike`/`toggleStar` 使用 `selectOne()` 检查记录是否存在再 `insert/delete`，属于Check-Then-Act反模式，并发下非原子→唯一索引冲突→DuplicateKeyException→500
2. Feign通知调用在@Transactional事务内同步执行，拉长DB行锁持有时间
3. 冗余selectOne查询增加一次DB往返

**优化方案**：
1. **DELETE-First原子模式**：先执行`DELETE WHERE post_id=? AND user_id=?`，根据affected rows判断当前状态（1=已取消，0=不存在需插入），消除selectOne
2. **DuplicateKeyException幂等处理**：INSERT冲突时catch异常并返回true（另一线程已插入），不抛500
3. **异步通知**：用`TransactionSynchronization.afterCommit()`+`CompletableFuture.runAsync()`在事务提交后异步发送通知，缩短持锁时间

**压测结果**（JMeter 50并发，60秒/接口，4核8G CentOS7）：
| 接口 | 错误率(前→后) | QPS(前→后) | 平均延迟 |
|------|-------------|-----------|---------|
| 点赞 | 38%→0% | 245→546/s (2.2x) | 40ms→83ms(注) |
| 收藏 | 37%→0% | 362→697/s (1.9x) | 42ms→65ms(注) |
| 下载(对照) | 0%→0% | - | 13ms→12ms |

注：平均延迟"升高"因优化前38%是快速失败的500响应拉低均值，优化后所有请求均成功走完整链路。

## [2026-07-03] 修复点击首页时页面在底部而非顶部的问题

### Bug修复

**问题**：每次通过导航栏点击进入首页（HomePage），页面都停在底部，应该从顶部开始。

**根因**：`useScrollRestoration` hook 为了接管滚动恢复，全局设置了 `window.history.scrollRestoration = 'manual'`，禁用了浏览器原生的路由切换自动滚动到顶部行为。对于不使用该 hook 的页面（如 HomePage），浏览器不会自动重置滚动位置，前一个页面的 scrollY 被带到新页面。例如用户在 SchoolDetailPage 滚动到底部后点击导航栏首页按钮，HomePage 就会停在底部。

**修复方案**：新增 [ScrollToTop](file:///e:/workspace_work/CampusShare/frontend/src/components/common/ScrollToTop.tsx) 组件，放在 BrowserRouter 内 Routes 之前。每次 pathname 变化时用 `useLayoutEffect`（浏览器绘制前）同步滚动到顶部，避免视觉抖动。

与 useScrollRestoration 的协作：
- 路由切换时 ScrollToTop 先滚动到 0
- 使用 useScrollRestoration 的页面在数据加载完成（ready=true）后由 hook 的 useLayoutEffect 覆盖，恢复到保存的精确位置
- 不使用 hook 的页面（如 HomePage）保持在顶部
- 仅监听 pathname 变化，不监听 hash 变化，PostDetailPage 的 #comment-xxx 锚点跳转不受影响

- `frontend/src/router/index.tsx`（引入 ScrollToTop，放在 Routes 前）

## [2026-07-03] 修复帖子列表页滚动恢复两个逻辑错误

### Bug修复

**问题1**：点进帖子详情再返回，没有正确还原进入时的滚动状态（如点进去时帖子在视口中间，返回后帖子出现在视口底部）。

**根因**：SchoolDetailPage 使用 IntersectionObserver 实现无限滚动。用户浏览时第二页及后续页面被自动加载，文档高度增大。保存的 scrollY 对应多页文档。返回时只加载第一页（文档较短），`scrollTo(0, savedY)` 被浏览器 clamp 到当前文档的最大滚动位置，导致帖子出现在视口底部而非中间。

**修复方案**：在 [useScrollRestoration.ts](file:///e:/workspace_work/CampusShare/frontend/src/hooks/useScrollRestoration.ts) 的恢复逻辑中添加 ResizeObserver 重试机制：
1. 恢复时如果保存的 Y 超出当前文档最大滚动位置，先滚动到最大位置以触发 IntersectionObserver 加载更多数据
2. 同时监听 body 尺寸变化，文档高度足够后自动恢复到精确位置
3. 5秒超时清理，避免内存泄漏
4. 引入 `restoringRef` 标志：恢复进行中阻止 scroll 事件覆盖 sessionStorage 中的原始值（避免触发 IntersectionObserver 产生的 scroll 事件写入错误值）

**问题2**：从帖子列表页返回到校园列表页，再重新进入同一学校时，滚动位置续上了之前的（应该从顶部开始）。

**根因**：返回校园列表页时 sessionStorage 中的滚动位置记录未被清除，重新进入时被错误恢复。

**修复方案**：导出 `clearScrollPosition(key)` 函数，在 SchoolDetailPage 返回按钮点击时调用，清除保存的滚动位置，使重新进入时从顶部开始。仅清除返回上级页面的场景，前进到帖子详情/用户主页时保留位置以便正确还原。

## [2026-07-03] 移除 AgentPage 进入页面时的平滑滚动动画

### Bug修复

**问题**：每次从导航栏点击进入 Agent 对话页时，聊天容器会执行一次从顶部滑到底部的平滑滚动动画（`scrollIntoView({ behavior: 'smooth' })`），用户感知为多余的视觉跳动。

**根因**：`AgentPage.tsx` 中负责自动滚动到底部的 useEffect 依赖 `[messages, streaming]`，无论首次挂载、加载历史会话、发送新消息还是流式输出，都统一使用 `behavior: 'smooth'`。首次进入/切换会话时由于 messages 从空/旧内容瞬间变为完整历史，浏览器需要平滑滚动较长距离，产生明显动画。

**修复方案**：引入 `shouldInstantScrollRef` 标志区分两类场景：

1. **瞬时滚动场景**（`behavior: 'auto'`，无动画）：
   - 组件首次挂载（welcome 消息或加载保存的会话）
   - `loadSession` 切换会话（成功/失败分支均设置标志）
   - `startNewChat` 新建对话
2. **保留平滑滚动场景**：用户发送新消息、AI 流式输出追加内容时仍使用 `behavior: 'smooth'`，跟随新内容平滑下滑

标志在 useEffect 中读取后立即重置为 `false`，确保后续消息更新回到平滑模式。

## [2026-07-03] 修复我的回复返回后滚动位置丢失：hook 根因修复

### Bug修复

**问题**：从「我的回复」点击进入帖子详情后返回，列表回到顶部，滚动位置丢失。此前的修复（commit c89134a）已覆盖 SchoolDetailPage/CategoryDetailPage，但 MyListPage 仍复现。

**根因分析**：`useScrollRestoration` hook 的两个设计缺陷叠加导致：

1. **cleanup 恢复 `scrollRestoration='auto'`**：组件卸载时 cleanup 把 `window.history.scrollRestoration` 从 `'manual'` 恢复为 `'auto'`。返回导航时浏览器在数据加载前就尝试原生滚动恢复，此时页面只有 loading spinner（高度不足），scrollY 被 clamp 到 0，触发的 scroll 事件把 sessionStorage 中已保存的真实位置覆盖为 0。

2. **加载期间 scroll 监听器仍活跃**：hook 在 mount 时就添加 scroll listener，不区分数据是否加载完成。加载期间的 scroll 事件（如浏览器 clamp 产生的）会覆盖已保存的真实位置。

**修复方案**（[useScrollRestoration.ts](file:///e:/workspace_work/CampusShare/frontend/src/hooks/useScrollRestoration.ts)）：

1. **不在 cleanup 中恢复 `scrollRestoration`**：保持 `'manual'` 不变，彻底阻止浏览器原生 scrollRestoration 在返回时干扰。组件卸载后该属性保持 manual，返回时浏览器不会尝试原生恢复，由 hook 的 useLayoutEffect 完全接管。
2. **`ready=false` 时不添加 scroll 监听器**：只有数据加载完成（`ready=true`）后才添加 scroll listener。加载期间页面高度不足产生的任何 scroll 事件都不会覆盖 sessionStorage。

此修复对所有使用该 hook 的页面（SchoolDetailPage、CategoryDetailPage、WarehousePage、MyListPage）均生效，是根因级修复。

## [2026-07-02] 修复 WebFlux 阻塞调用导致 RAG 知识库索引全部失败

### 业务背景（被动场景型 - 系统驱动）

**Why now**: Week 2 RAG 模块 1-11 代码已完成并部署，但触发 `POST /internal/agent/knowledge/reindex` 时 18 篇文档全部 failed（`inserted:0, failed:18, vectorCount:0`），导致 RAG 检索无结果，agent 回答"怎么登录注册"时仍出现 LLM 幻觉（说"微信登录/企业微信登录"等不相关内容）。

**What if not done**: 知识库向量为 0 → RAG 检索上下文为空 → agent 回答完全依赖 LLM 常识 → 幻觉持续，用户无法通过 agent 获取项目真实功能信息，Week 2 RAG 投入完全无效。

**What success means**: reindex 返回 `inserted:18, failed:0, vectorCount:18`，问"怎么登录注册"时 agent 准确回答"手机号和邮箱两种登录方式"。

### 问题

1. **根因**：`InternalAgentController.reindexKnowledge()` 返回同步类型 `Result<Map>`，在 WebFlux 的 reactor-http-epoll 事件循环线程上执行
2. **`KnowledgeIngestionService.ingestAll()` 内部调用 `embeddingClient.embed(...).block()`** → WebFlux 禁止在事件循环线程上调用 `.block()`，直接抛 `IllegalStateException`
3. 18 篇文档每篇都因 `IllegalStateException` 失败，embedding API 从未被调用
4. `PostVectorService.syncAll()` 和 `syncPost()` 同样缺少 `.subscribeOn(Schedulers.boundedElastic())`，帖子向量同步也会失败
5. **诊断弯路**：初期误判为 .env 文件 Unicode 字符污染（xxd hex dump 证实 .env 干净，反引号是终端渲染假象），实际是代码层面的线程模型错误

### 解决方案

将所有阻塞调用（JDBC、`.block()`、Feign）用 `Mono.fromCallable().subscribeOn(Schedulers.boundedElastic())` 包裹，移到专用弹性线程池执行，不阻塞 reactor 事件循环：

1. **InternalAgentController**：3 个端点从同步 `Result<T>` 改为 `Mono<Result<T>>` + `fromCallable + subscribeOn(boundedElastic)`
   - `reindexKnowledge()`: 知识库重建索引
   - `knowledgeStatus()`: 知识库向量计数
   - `vectorStatus()`: 向量库总体状态

2. **PostVectorService**：2 个方法补加 `.subscribeOn(Schedulers.boundedElastic())`
   - `syncAll()`: `Mono.fromCallable(this::doSyncAll)` 内部有 `embedBatch().block()` + Feign + JDBC
   - `syncPost()`: 两个分支（DELETE 的 `fromRunnable` + 非 DELETE 的 `fromCallable`）都有阻塞调用

3. **AgentChatService 无需修改**：`prepareContext()` 已正确包裹在 `Mono.fromCallable().subscribeOn(boundedElastic())` 中，chat 的 RAG 检索 `.block()` 是安全的

### 技术细节

- **WebFlux 线程模型**：reactor-http-epoll-N 是事件循环线程，禁止阻塞操作。`.block()` 会抛 `IllegalStateException: block()/blockFirst()/blockLast() are blocking, which is not supported in thread reactor-http-epoll-1`
- **boundedElastic 线程池**：专为阻塞任务设计的弹性线程池，适合 JDBC、`.block()`、Feign 同步调用
- **KnowledgeScheduler 无需修改**：`onStartup()` 用 `new Thread()`、`scheduledIngestion()` 用 `@Scheduled`，都不是 reactor 线程，`.block()` 合法
- **PostVectorScheduler 同理**：调度器线程上 `.block()` 合法
